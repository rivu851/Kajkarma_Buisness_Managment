1. PROJECT OVERVIEW & ARCHITECTURE
================================================================================

  KajKarma BMT is a multi-role business management backend for managing:
    - Sales pipeline (leads → clients → communications)
    - Projects & employee worklogs
    - HR (employees, salaries, reimbursements)
    - Finance (revenues, payments, subscriptions, upcoming payments)
    - Reports
    - Automated reminder & notification system
    - Role-based dashboards

  Entry point:   src/server.ts   (bootstraps DB, seeds roles/admin, starts crons)
  App config:    src/app.ts      (Express setup, middleware, routes)
  Routes index:  src/routes/index.ts
  All routes mounted under: /api/v1

  Key directories:
    src/routes/         — Express routers
    src/controllers/    — HTTP handlers (req → service → res)
    src/services/       — Business logic
    src/repositories/   — MongoDB queries
    src/models/         — Mongoose schemas
    src/middlewares/    — auth, permission, validation, error, rate-limit
    src/validations/    — Zod schemas for every route
    src/constants/      — enums.ts, permissions.ts, roles.ts
    src/utils/          — jwt, encryption, recordScope, reminderScope, AppError
    src/cron/           — Scheduled background jobs
    src/config/         — env.ts, db.ts, logger.ts

3. SERVER STARTUP & BOOT SEQUENCE
================================================================================

  Order of operations when server starts:

  1. connectDB()             — Connect to MongoDB via MONGODB_URI
  2. seedSystemRoles()       — Ensure all 7 system roles exist with correct
                               default permissions. If a system role already
                               exists, its permissions are synced/updated.
  3. seedSuperAdmin()        — If NO users exist in the DB, create the first
                               super_admin user using SUPER_ADMIN_EMAIL +
                               SUPER_ADMIN_PASSWORD. Does NOT run on subsequent
                               restarts if users already exist.
  4. runScheduledReminderSync() — Run full reminder sync once at boot
                               (catches any drift since last run).
  5. startReminderCron()     — Start cron job: every 5 minutes
  6. startReminderDigestCron() — Start cron job: daily at 08:00 IST
  7. app.listen(PORT)        — Start HTTP server

  Health check endpoint (no auth needed):
    GET /health  →  { status: "ok", timestamp: "..." }


================================================================================
4. GLOBAL MIDDLEWARE STACK
================================================================================

  Applied to ALL routes in this order:
  1. helmet()           — Sets security HTTP headers (XSS protection, HSTS, etc.)
  2. cors()             — Allows requests from CORS_ORIGIN only, with credentials
  3. compression()      — gzip response compression
  4. express.json()     — Parses JSON bodies up to 10mb
  5. express.urlencoded() — Parses URL-encoded bodies
  6. cookieParser()     — Parses signed cookies
  7. globalRateLimit    — In-memory rate limit: RATE_LIMIT_MAX requests per
                          RATE_LIMIT_WINDOW_MS (defaults: 100 req / 15 min)
                          Returns 429 if exceeded.
  8. routes             — All /api/v1/* routes
  9. 404 handler        — { success: false, message: "Route not found" }
 10. errorMiddleware    — Global error handler (last)

  Per-route middleware (applied selectively inside routers):
  - authenticate        — Verifies Bearer JWT token (401 if missing/invalid)
  - requireRole(...)    — Checks user.roleName against allowed role names (403)
  - requirePermission(module, action)  — Checks role permissions + access control
                          overrides for the given module:action pair (403)
  - validate(zodSchema) — Validates request body/params/query against Zod schema
                          Returns 422 with field errors on validation failure

5. AUTHENTICATION SYSTEM
================================================================================

  Token Format:
    - Authorization header: "Bearer <access_token>"
    - Access token:   JWT, expires in JWT_ACCESS_EXPIRES_IN (default 15m)
    - Refresh token:  JWT, expires in JWT_REFRESH_EXPIRES_IN (default 7d)

  JWT Payload (decoded into req.user after authentication):
    {
      userId:   string   (MongoDB ObjectId of user)
      email:    string
      roleName: string   (e.g. "super_admin", "hr")
      roleId:   string   (MongoDB ObjectId of role)
    }

  Authentication Flow:
    1. Client POSTs to /api/v1/auth/login with email + password
    2. Server validates credentials, checks user.status === 'active'
    3. Creates LoginAudit record (IP, user-agent, success/failure)
    4. Returns { accessToken, refreshToken, user: {...} }
    5. Client attaches accessToken in Authorization header for every request
    6. When access token expires, client POSTs refreshToken to /api/v1/auth/refresh
    7. Server issues new accessToken + refreshToken pair

  Conditions that block login:
    - User not found                → 401 "Invalid credentials"
    - Password does not match       → 401 "Invalid credentials"
    - user.status !== 'active'      → 403 "Account is not active"

  All LoginAudit records are stored in the login_audits collection.

6. ROLE SYSTEM & HIERARCHY
================================================================================

  There are 7 SYSTEM roles (cannot be deleted, names cannot be changed):

  Role Name          Hierarchy Level   Description
  ─────────────────────────────────────────────────────────────────────────────
  super_admin              100         Full unrestricted access to everything
  admin                     80         Almost full access (see permission matrix)
  sales_manager             60         CRM: leads, clients, communications
  project_manager           60         Projects, employees (read), worklogs, reports
  hr                        60         Employees, salaries, reimbursements
  finance                   60         Revenue, payments, salary, subscriptions
  sales_executive           40         Limited CRM access, own records only

  Custom roles can be created by super_admin with custom permission sets.

  Role Hierarchy Rules:
    - A user can only assign roles with a LOWER hierarchy level than their own.
    - Example: admin (80) can assign up to level 79 — can assign sales_manager (60)
      but CANNOT assign another admin (80) or super_admin (100).
    - A user cannot change their own role.
    - A user cannot delete themselves.

  Super Admin Bypass:
    - The requirePermission() middleware short-circuits for super_admin.
    - super_admin always passes ALL permission checks regardless of stored
      permissions.

7. PERMISSION MATRIX (All Roles × All Modules)
================================================================================

  Format: R=read, C=create, U=update, D=delete, -=no access, ALL=full RCUD

  users               ALL          ALL        -          -           -         -      -
  roles               ALL          R          -          -           -         -      -
  leads               ALL          ALL        ALL        R,C,U       -         -      -
  clients             ALL          ALL        ALL        R,C,U       R         -      -
  communications      ALL          ALL        ALL        ALL         -         -      -
  reminders           ALL          ALL        ALL        ALL         ALL       ALL    ALL
  projects            ALL          ALL        -          R           ALL       -      -
  employees           ALL          R,C,U      -          -           R         ALL    R
  worklogs            ALL          ALL        -          R,C,U       R,C,U     -      -
  revenue             ALL          R          -          -           -         -      ALL
  payments            ALL          R          -          -           -         -      ALL
  salary              ALL          R          -          -           -         ALL    ALL
  reimbursements      ALL          ALL        -          -           -         ALL    ALL
  reports             ALL          ALL        R          -           ALL       -      -
  subscriptions       ALL          R          -          -           -         -      ALL
  dashboard           ALL          R          R          R           R         R      R
  forecasting         ALL          R          -          -           -         -      -

  Notes:
  - admin cannot DELETE employees (R,C,U only)
  - sales_executive cannot DELETE leads or clients (R,C,U only)
  - sales_executive and project_manager cannot DELETE worklogs (R,C,U only)
  - All modules not listed for a role default to NO ACCESS
  - super_admin bypasses the permission check entirely (no DB query needed)
  - Access control overrides (set by super_admin per user) take priority over
    role defaults

8. PERMISSION OVERRIDE SYSTEM (Per-User Access Control)
================================================================================

  super_admin can set custom module permissions for any individual user.
  These overrides are stored in the access_controls collection.

  How it works:
    1. requirePermission() middleware fetches:
         a) The user's role permissions (from roles collection)
         b) The user's access_control overrides (from access_controls collection)
    2. If an override exists for the module → use override
    3. If no override → fall back to role permissions
    4. If neither grants the action → 403

  This allows super_admin to:
    - Grant a finance user access to leads (not in their role)
    - Restrict an admin from accessing certain modules
    - Give a sales_executive delete permission on leads

  Endpoints for this:
    GET  /api/v1/users/:id/access-control   → View overrides for a user
    PATCH /api/v1/users/:id/access-control  → Set overrides for a user
  Both require: super_admin only

9. RECORD-LEVEL DATA SCOPING
================================================================================

  Beyond module permissions, some roles have record-level restrictions — they
  can read the module but only see their own assigned records.

  Who has FULL access (sees everything in a module):
    super_admin, admin, sales_manager, project_manager, hr, finance

  Sales Executive scoping:
    - leads:          Only sees leads where assigned_user_id === their userId
    - clients:        Only sees clients where assigned_manager_id === their userId
    - communications: Only sees communications where user_id === their userId
    - projects:       READ-ONLY, sees all projects (no record scope restriction)
    - worklogs:       Only sees worklogs where employee's user_id === their userId

  Employee-level scoping (for project/worklog access):
    - Worklogs: HR/Admin/Project Manager see all; otherwise only own worklogs

  Reminder scoping:
    - super_admin, admin:    See ALL reminders
    - sales_manager:         Sees reminders for modules: leads, clients,
                             communications, upcoming-payments
    - project_manager:       Sees reminders for modules: projects, reports,
                             worklogs
    - hr:                    Sees reminders for modules: salary, reimbursements,
                             employees
    - finance:               Sees reminders for modules: salary, upcoming-payments,
                             subscriptions, reimbursements, revenue
    - sales_executive + others: Only sees reminders assigned to themselves

  /reminders/my — always returns ONLY reminders assigned to the requesting user,
                  regardless of role.

11. MODULE: AUTH  →  /api/v1/auth
================================================================================

  No authentication required for login and refresh.
  Rate limiting applies globally.

  ────────────────────────────────────────────────────────────────────────────
  POST /api/v1/auth/login
  ────────────────────────────────────────────────────────────────────────────
  Access:   Public (no auth needed)
  Body: {
    email:    string (required, valid email format)
    password: string (required, min 1 char)
  }
  Returns:  { accessToken, refreshToken, user: { _id, name, email, role } }
  Logic:
    - Finds user by email
    - Compares password with bcrypt (12 rounds)
    - Checks user.status === 'active' (403 if not)
    - Creates LoginAudit record with IP, user-agent, timestamp, success/failure
    - Returns JWT access token (15m) + refresh token (7d)

  ────────────────────────────────────────────────────────────────────────────
  POST /api/v1/auth/logout
  ────────────────────────────────────────────────────────────────────────────
  Access:   Authenticated (any role)
  Body:     none
  Returns:  { message: "Logged out successfully" }
  Logic:
    - Clears any auth cookies
    - Does NOT invalidate tokens server-side (stateless JWT)
    - Client should discard tokens

  ────────────────────────────────────────────────────────────────────────────
  POST /api/v1/auth/refresh
  ────────────────────────────────────────────────────────────────────────────
  Access:   Public (no auth needed, but requires valid refresh token)
  Body: {
    refreshToken: string (required)
  }
  Returns:  { accessToken, refreshToken }
  Logic:
    - Verifies the refresh token signature and expiry
    - Fetches current user from DB to get latest role/status
    - Checks user.status === 'active' (403 if not)
    - Issues new access token + refresh token pair

  ────────────────────────────────────────────────────────────────────────────
  GET /api/v1/auth/me
  ────────────────────────────────────────────────────────────────────────────
  Access:   Authenticated (any role)
  Body:     none
  Returns:  { user: { _id, name, email, role, status, ... } }
  Logic:
    - Returns the currently authenticated user's full profile
    - Password field is always removed from response

12. MODULE: USERS  →  /api/v1/users
================================================================================

  All routes require authentication.
  Most require admin or super_admin role.

  ────────────────────────────────────────────────────────────────────────────
  GET /api/v1/users
  ────────────────────────────────────────────────────────────────────────────
  Access:   admin or super_admin
  Query params:
    page:    number (default: 1)
    limit:   number (default: 20, max: 100)
    search:  string (optional — searches name and email)
  Returns:  Paginated list of users with role info
  Logic:    Lists all users. Passwords excluded from response.

  ────────────────────────────────────────────────────────────────────────────
  POST /api/v1/users
  ────────────────────────────────────────────────────────────────────────────
  Access:   admin or super_admin
  Body: {
    name:     string (required, min 2 chars)
    email:    string (required, valid email)
    password: string (required, min 8 chars)
    role_id:  string (required, must be a valid role ObjectId)
    source:   string (optional)
  }
  Returns:  Created user object (without password)
  Conditions:
    - Email must be unique across all users (409 if duplicate)
    - The role being assigned must have a hierarchy level LOWER than the
      requestor's own level. Example: an admin (level 80) cannot create a user
      with the super_admin role (level 100).
    - Password is hashed with bcrypt (12 rounds) before storage

  ────────────────────────────────────────────────────────────────────────────
  GET /api/v1/users/:id
  ────────────────────────────────────────────────────────────────────────────
  Access:   admin or super_admin
  Params:   id (MongoDB ObjectId of user)
  Returns:  Single user object with role info (password excluded)

  ────────────────────────────────────────────────────────────────────────────
  PATCH /api/v1/users/:id
  ────────────────────────────────────────────────────────────────────────────
  Access:   admin or super_admin
  Params:   id
  Body: {
    name:   string (optional, min 2 chars)
    email:  string (optional, valid email)
    source: string (optional)
  }
  Returns:  Updated user object
  Conditions:
    - Only name, email, source can be updated via this endpoint
    - Email uniqueness check still applies

  ────────────────────────────────────────────────────────────────────────────
  DELETE /api/v1/users/:id
  ────────────────────────────────────────────────────────────────────────────
  Access:   super_admin ONLY
  Params:   id
  Returns:  { message: "User deleted" }
  Conditions:
    - CANNOT delete your own account (403 "Cannot delete your own account")
    - Hard delete from database

  ────────────────────────────────────────────────────────────────────────────
  PATCH /api/v1/users/:id/status
  ────────────────────────────────────────────────────────────────────────────
  Access:   admin or super_admin
  Params:   id
  Body: {
    status: "active" | "inactive" | "suspended"  (required)
  }
  Returns:  Updated user object
  Conditions:
    - CANNOT change your own status (403 "Cannot change your own status")
    - Setting status to inactive/suspended will block that user from logging in

  ────────────────────────────────────────────────────────────────────────────
  PATCH /api/v1/users/:id/role
  ────────────────────────────────────────────────────────────────────────────
  Access:   admin or super_admin
  Params:   id
  Body: {
    role_id: string (required, valid role ObjectId)
  }
  Returns:  Updated user object
  Conditions:
    - The new role must have hierarchy level LOWER than the requestor
    - Cannot change your own role

  ────────────────────────────────────────────────────────────────────────────
  GET /api/v1/users/:id/permissions
  ────────────────────────────────────────────────────────────────────────────
  Access:   admin or super_admin
  Params:   id
  Returns:  The effective permissions for this user
            (role permissions merged with any access control overrides)

  ────────────────────────────────────────────────────────────────────────────
  GET /api/v1/users/:id/access-control
  ────────────────────────────────────────────────────────────────────────────
  Access:   super_admin ONLY
  Params:   id
  Returns:  The per-user access control override document (or empty if none set)

  ────────────────────────────────────────────────────────────────────────────
  PATCH /api/v1/users/:id/access-control
  ────────────────────────────────────────────────────────────────────────────
  Access:   super_admin ONLY
  Params:   id
  Body: {
    module_permissions: {
      "<module_name>": {
        read:   boolean,
        create: boolean,
        update: boolean,
        delete: boolean
      },
      ...
    }
  }
  Returns:  Updated access control document
  Logic:
    - Replaces the per-user module permission overrides
    - These overrides take precedence over role-level permissions
    - Module names must be valid (from MODULES constant)
    - Pass null/omit module_permissions to clear all overrides

  ────────────────────────────────────────────────────────────────────────────

14. MODULE: LEADS  →  /api/v1/leads
================================================================================

  All routes require authentication + reminders:read or leads:read/write permission.

  Lead Stages (in order):
    new → contacted → proposal_sent → negotiation → won / lost

  Lead Statuses: active | on_hold | unresponsive | dropped

  Record Scoping:
    - sales_executive: Only sees leads where assigned_user_id === their userId
    - All other roles with leads:read: See all leads

  ────────────────────────────────────────────────────────────────────────────
  GET /api/v1/leads
  ────────────────────────────────────────────────────────────────────────────
  Access:   leads:read permission required
  Query params:
    page:             number (default: 1)
    limit:            number (default: 20, max: 100)
    search:           string (searches lead_name, email, company_name)
    stage:            new | contacted | proposal_sent | negotiation | won | lost
    status:           active | on_hold | unresponsive | dropped
    assigned_user_id: ObjectId (filter by assigned user)
    source:           string
  Returns:  Paginated list of leads (scoped by role — sales_exec sees only own)

  ────────────────────────────────────────────────────────────────────────────
  POST /api/v1/leads
  ────────────────────────────────────────────────────────────────────────────
  Access:   leads:create permission required
  Body: {
    lead_name:        string (required)
    phone_number:     string (optional)
    email:            string (optional, valid email or "")
    company_name:     string (optional)
    sector:           string (optional)
    source:           string (required)
    stage:            lead stage enum (optional, default: "new")
    status:           lead status enum (optional, default: "active")
    tags:             string[] (optional)
    notes:            string (optional)
    follow_up_date:   date (optional)
    assigned_user_id: ObjectId (optional)
  }
  Returns:  Created lead document
  Side effects:

18. MODULE: EMPLOYEES  →  /api/v1/employees
================================================================================

  Employee Statuses: active | on_leave | resigned | terminated

  Sensitive Fields (restricted based on viewer's role):
    salary, pending_salary, date_of_birth, bank_account_holder,
    bank_name, account_number, ifsc_code, branch_name, upi_id

  Who can see sensitive fields:
    - super_admin, admin, hr, finance: Always visible

    - account_number is shown as masked (e.g. "****1234") to everyone
      except HR/Finance/super_admin who see the real value

    status:     active | on_leave | resigned | terminated
    department: string
  Returns:  Paginated list of employees (sensitive fields filtered by role)

  POST /api/v1/employees
  ────────────────────────────────────────────────────────────────────────────
  Access:   employees:create permission
  Body: {
    user_id:           ObjectId (optional — links employee to a system user account)
    full_name:         string (required)
    date_of_birth:     date (optional)
    phone_number:      string (optional)
    email:             string (optional, valid email or "")
    department:        string (required)
    role_designation:  string (required)
    joining_date:      date (required)
    salary:            number (optional, min 0)
    pending_salary:    number (optional, min 0)
    status:            employee status enum (optional, default: "active")
    -- bank fields (optional) --
    bank_account_holder: string
    bank_name:           string
    account_number:      string (encrypted before storage)
    ifsc_code:           string (encrypted before storage)
    branch_name:         string
    upi_id:              string (encrypted before storage)
  }
  Returns:  Created employee document
  Conditions:
    - user_id must be unique per employee (sparse unique index — one employee
      record per user account)
    - Bank fields (account_number, upi_id) are encrypted via AES-256-GCM

  ────────────────────────────────────────────────────────────────────────────
  GET /api/v1/employees/me
  ────────────────────────────────────────────────────────────────────────────

  Returns:  The employee record linked to the current user (via user_id)
            Returns full record including sensitive fields
            Returns 404 if no employee linked to this user account

28. MODULE: REMINDERS  →  /api/v1/reminders
================================================================================

  Reminder Types:     lead_followup | client_payment | project_deadline |
                      salary_due | report_upload | subscription_renewal |
                      reimbursement_review | communication_followup | custom
  Reminder Statuses:  pending | done | snoozed | rescheduled | cancelled
  Reminder Priorities: low | medium | high | critical

  Soft Delete: Reminders use deleted_at field. Deleted reminders are excluded
               from all queries but remain in the DB for audit.

  Dedup Key: Auto-generated reminders use a dedup_key to prevent duplicates.
             Format: "<type>|<module>|<record_id>|<date_or_suffix>"
             Example: "lead_followup|leads|64abc123...|2026-06-10"

  Access Control for individual reminder:
    - The assignee (assigned_user_id === userId) can always access
    - super_admin and admin can access all reminders
    - Role managers can access reminders for their module scope:
        sales_manager: leads, clients, communications, upcoming-payments
        project_manager: projects, reports, worklogs
        hr: salary, reimbursements, employees
        finance: salary, upcoming-payments, subscriptions, reimbursements, revenue

  Write Access (PATCH operations on individual reminder):
    - Only the ASSIGNEE can modify their own reminder
    - super_admin and admin can modify any reminder
    - Other role managers can READ but NOT WRITE other people's reminders

  Audit History:
    - Every action (created, updated, read, unread, done, snoozed, rescheduled,
      deleted) is logged in reminder.audit_history[]
    - audit_history is only visible to super_admin in GET /:id response

29. CRON JOBS (Background Tasks)
================================================================================

  ┌─────────────────────────────────────────────────────────────────────────┐
  │ Reminder Sync Cron                                                       │
  │ Schedule: Every 5 minutes  (*/5 * * * *)                                 │
  │ Source:   src/cron/reminder.cron.ts                                      │
  └─────────────────────────────────────────────────────────────────────────┘
  Runs runScheduledReminderSync() which does ALL of the following in parallel:

  1. syncLeadReminders()
     - Finds all active leads (status=active, stage NOT won/lost) with
       follow_up_date set
     - Creates/updates lead_followup reminders
     - Priority: "high" if overdue, "medium" otherwise

  2. syncProjectReminders()
     - Finds projects with end_date set, status in [not_started, in_progress, on_hold]
     - Only triggers when deadline is ≤7 days away
     - Priority logic:
         daysLeft < 0:  title = "Project deadline missed"  → CRITICAL
         daysLeft = 0:  title = "Project deadline is today" → CRITICAL
         daysLeft ≤ 3:  title = "Project deadline in N days" → HIGH
         daysLeft ≤ 7:  title = "Project deadline in N days" → MEDIUM

  3. syncPaymentReminders()
     - Finds upcoming payments with status in [pending, overdue]
     - Creates/updates client_payment reminders
     - Priority: "critical" if overdue, "high" otherwise
     - Assigned to: assigned_follow_up_user or created_by

  4. syncSubscriptionReminders()
     - Finds active/expiring_soon subscriptions with renewal_date set
     - Only triggers when ≤30 days to renewal
     - Priority logic:
         daysLeft < 0:  "Subscription expired" → CRITICAL
         daysLeft = 0:  "Subscription renewal is today" → CRITICAL
         daysLeft ≤ 7:  "Subscription renewing in N days" → HIGH
         daysLeft ≤ 30: "Subscription renewing in N days" → MEDIUM

  5. syncSalaryReminders()
     - Triggers only AFTER end-of-previous-month
     - Finds salary records with status = "pending"
     - Creates CRITICAL salary_due reminders for Finance + Super Admin users

  6. syncReimbursementReminders()
     - Finds reimbursements in [submitted, under_review]
     - Creates reimbursement_review reminders for all HR and Finance users (HIGH)
     - If amount >= REIMBURSEMENT_APPROVAL_THRESHOLD:
         Also creates CRITICAL reminder for all Super Admin users

  7. syncCommunicationReminders()
     - Finds communications with next_follow_up_date set
     - Creates communication_followup reminders for the communication's user_id

  8. syncReportUploadReminders()
     - ONLY runs on the 1st day of the month
     - Checks in_progress/completed projects that have NO report uploaded
       for the previous month
     - Creates report_upload reminders (MEDIUM) for project.created_by

  After all syncs:
  9. cancelStaleSourceReminders()
     - Cancels pending reminders for won/lost leads
     - Cancels pending reminders for received/cancelled upcoming payments
     - Cancels pending reminders for completed/cancelled projects
     - Cancels pending reminders for expired/cancelled subscriptions
     - Cancels orphaned communication reminders (no active follow-up date)

  10. wakeSnoozedReminders()
      - Sets all snoozed reminders back to "pending" when snoozed_until <= now

  11. escalateOverdueReminders()
      - Finds pending/rescheduled reminders where reminder_date < today
        AND priority in [low, medium, high]
      - Sets their priority to "critical"

  ┌─────────────────────────────────────────────────────────────────────────┐
  │ Daily Reminder Digest Cron                                               │
  │ Schedule: Daily at 08:00 Asia/Kolkata (IST)  (0 8 * * *)                │
  │ Source:   src/cron/reminder-digest.cron.ts                               │
  └─────────────────────────────────────────────────────────────────────────┘
  Runs runDailyReminderDigest() which:
    - Gathers today's pending + overdue reminders for all active users
    - Sends a digest email to DAILY_REMINDER_DIGEST_EMAIL (if SMTP configured)
    - Logs to DailyDigestLog collection per user

30. AUTO-REMINDER SYSTEM (Trigger Conditions)
================================================================================

  Auto-reminders are created/updated by the system at these trigger points.
  All use dedup_key to prevent duplicate reminders.
  Status "done" or "cancelled" reminders are NOT overwritten.

  TRIGGER 1: Lead follow-up date set or changed
  ─────────────────────────────────────────────
  When:   PATCH /leads/:id (follow_up_date changes)
          POST /leads (follow_up_date in body)
  Action: Cancel existing lead_followup reminders for this lead
          Create new lead_followup reminder
  Key:    lead_followup|leads|{leadId}|{YYYY-MM-DD}
  Assigned to: lead.assigned_user_id
  Priority: "high" if date is in the past, "medium" otherwise

  TRIGGER 2: Lead closed (won or lost)
  ────────────────────────────────────
  When:   PATCH /leads/:id/stage with stage = "won" or "lost"
  Action: Cancel ALL pending lead_followup reminders for this lead

  TRIGGER 3: Lead converted to client
  ────────────────────────────────────
  When:   POST /leads/:id/convert
  Action: Cancel ALL pending reminders for this lead

  TRIGGER 4: Communication follow-up date set
  ───────────────────────────────────────────
  When:   POST /communications (next_follow_up_date in body)
          PATCH /communications/:id (next_follow_up_date changes)
  Action: Cancel existing communication_followup reminders for this comm
          Create new communication_followup reminder
  Key:    communication_followup|communications|{commId}|{YYYY-MM-DD}
  Assigned to: communication.user_id (who logged the communication)
  Priority: "medium"

  TRIGGER 5: Upcoming payment created
  ────────────────────────────────────
  When:   POST /upcoming-payments
  Action: If reminder_date AND assigned_follow_up_user → create client_payment
          reminder
  Key:    client_payment|upcoming-payments|{paymentId}|main
  Priority: "critical" if due_date already past, "high" otherwise

  TRIGGER 6: Upcoming payment received or cancelled
  ──────────────────────────────────────────────────
  When:   PATCH /upcoming-payments/:id/receive
          PATCH /upcoming-payments/:id/cancel
  Action: Cancel ALL pending client_payment reminders for this payment

  TRIGGER 7: Project completed or cancelled
  ─────────────────────────────────────────
  When:   PATCH /projects/:id/status with status = "completed" or "cancelled"
  Action: Cancel ALL pending project_deadline reminders for this project

  TRIGGER 8: Reimbursement submitted
  ────────────────────────────────────
  When:   POST /reimbursements
  Action: Create reimbursement_review reminders for all HR + Finance users (HIGH)
          If amount >= REIMBURSEMENT_APPROVAL_THRESHOLD:
            Create CRITICAL reminders for all Super Admin users
  Key:    reimbursement_review|reimbursements|{rId}|{approverId}
          reimbursement_review|reimbursements|{rId}|sa|{adminId}

  PRIORITY ESCALATION (Cron):
  When:   Every 5 minutes via cron
  Action: Any pending/rescheduled reminder where reminder_date < today
          AND priority is low/medium/high → upgraded to "critical"

32. ERROR HANDLING & RESPONSE FORMAT
================================================================================

  All successful responses follow this format:
  {
    "success": true,
    "message": "...",
    "data": { ... }  or  [ ... ]  or  { items: [...], total: N, page: N, limit: N }
  }

  All error responses follow this format:
  {
    "success": false,
    "message": "Human-readable error message",
    "errors": [ ... ]  (optional, for validation errors)
  }

  HTTP Status Codes Used:
  ──────────────────────
  200 OK                    Successful GET, PATCH, DELETE
  201 Created               Successful POST (resource created)
  400 Bad Request           Business logic violation (e.g. lead not won)
  401 Unauthorized          Missing or invalid authentication token
  403 Forbidden             Valid token but insufficient permissions
  404 Not Found             Resource not found
  409 Conflict              Duplicate key (email, salary month/year, etc.)
  422 Unprocessable Entity  Zod validation failed (with errors array)
  429 Too Many Requests     Rate limit exceeded
  500 Internal Server Error Unexpected server error

  Specific error cases:
  ─────────────────────
  401 "Authentication token required"   — No Bearer token in header
  401 "Invalid or expired token"        — Token expired or tampered
  401 "Invalid credentials"             — Wrong email or password
  403 "Account is not active"           — User status is inactive/suspended
  403 "Access denied: role not permitted"    — requireRole check failed
  403 "Access denied: insufficient permission for <module>:<action>"
                                        — requirePermission check failed
  403 "Cannot delete your own account"  — Tried to DELETE self
  403 "Cannot change your own status"   — Tried to PATCH own status
  403 "Only the assignee can modify this reminder"
  400 "Lead must be in 'won' stage to convert"
  400 "Cannot set status to in_progress without assigned employees and end date"
  409 "Email already exists"
  409 "Salary record already exists for this employee, month and year"

  In development mode: full error stack trace is returned in the response.
  In production mode: only the message is returned (no internals leaked).

33. SECURITY FEATURES SUMMARY
================================================================================

  1. JWT Authentication
     - Stateless Bearer tokens, no sessions
     - Access token: 15 minutes
     - Refresh token: 7 days
     - Separate secrets for access and refresh tokens

  2. Role-Based Access Control (RBAC)
     - 7 system roles with fixed permissions
     - Custom roles supported
     - requireRole() checks exact role names
     - requirePermission() checks module:action granularity

  3. Per-User Permission Overrides
     - super_admin can grant/revoke specific module:actions per user
     - Overrides take priority over role defaults

  4. Record-Level Scoping
     - sales_executive only sees their own leads/clients/communications
     - Worklog access limited to own employee record for lower roles

  5. Data Encryption
     - AES-256-GCM encryption for employee bank fields (account_number, upi_id)
     - Random IV per field per encrypt operation
     - ENCRYPTION_KEY stored as environment variable (never in code)
     - Masked display (****1234) for non-privileged users

  6. Password Security
     - bcrypt with cost factor 12
     - Password never returned in any response

  7. Rate Limiting
     - 100 requests per 15 minutes globally
     - In-memory (per-process) — for production, use Redis-backed limiter

  8. HTTP Security Headers (Helmet)
     - Content-Security-Policy
     - X-XSS-Protection
     - X-Frame-Options
     - Strict-Transport-Security
     - X-Content-Type-Options

  9. CORS
     - Restricted to CORS_ORIGIN (default: localhost:3000)
     - Credentials allowed

  10. Input Validation (Zod)
      - All request bodies, params, and query strings validated
      - Invalid inputs rejected at middleware level before reaching controllers

  11. Audit Logging
      - Login attempts (success/failure) logged with IP + user agent
      - All reminder actions logged with actor, timestamp, old/new values
